# 数据库表结构说明

本文档详细说明学生选课系统中各数据库表的作用和设计意图。

## 数据库信息

- **数据库名称**: `app_stu_select`
- **字符集**: `utf8mb4`
- **初始化脚本**: `app_stu_select-Mysql.sql`

---

## 1. tb_admin (管理员表)

**作用**: 存储系统管理员账号信息，管理员拥有系统最高权限，负责维护课程、教师、学生、选课记录等核心数据。

**主要字段**:
- `id`: 主键，19位数字字符串（时间戳生成）
- `username`: 登录用户名，唯一索引
- `password`: BCrypt 加密的密码哈希值（强度10）
- `name`: 管理员真实姓名
- `tele`: 联系电话

**业务特点**:
- 管理员可以执行所有 CRUD 操作
- 所有管理员操作会被记录到审计日志表
- 支持多个管理员账号，可分配不同职责（如系统管理员、审计管理员）

---

## 2. tb_teacher (教师表)

**作用**: 存储教师账号和个人信息，教师主要负责课程授课和成绩录入。

**主要字段**:
- `id`: 主键
- `username`: 登录用户名，唯一索引
- `password`: BCrypt 加密的密码哈希值
- `numb`: 教工号，用于学校内部标识
- `tname`: 教师姓名
- `tbirthday`: 出生日期
- `tposition`: 职位/职称（如教授、副教授、讲师）
- `ttel`: 联系电话
- `age`: 年龄
- `gender`: 性别

**业务特点**:
- 教师只能查看和管理自己授课的课程
- 教师只能为自己负责的课程录入成绩
- 教师可以查看选修自己课程的学生信息
- 教师可以维护个人资料

---

## 3. tb_student (学生表)

**作用**: 存储学生账号和个人信息，学生是选课系统的主要使用者。

**主要字段**:
- `id`: 主键
- `username`: 登录用户名，唯一索引
- `password`: BCrypt 加密的密码哈希值
- `numb`: 学号，学生在学校的唯一标识
- `sname`: 学生姓名
- `sdept`: 所属学院
- `sbirthday`: 出生日期
- `tele`: 联系电话
- `email`: 电子邮箱
- `ssex`: 性别
- `age`: 年龄
- `smajor`: 专业
- `sclass`: 班级

**业务特点**:
- 学生可以自助注册账号
- 学生在时间窗口内可以选课和退课
- 学生只能查看自己的选课记录和成绩
- 学生可以查询 GPA 和成绩统计信息
- 学生可以维护个人资料

---

## 4. tb_course (课程表)

**作用**: 存储课程基本信息，是选课系统的核心数据实体。

**主要字段**:
- `id`: 主键
- `name`: 课程名称
- `score`: 学分（用于 GPA 计算）
- `numb`: 课程编号，唯一索引
- `tid`: 任课教师 ID（外键关联 tb_teacher.id）
- `jianjie`: 课程简介
- `dept`: 开课学院
- `max_students`: 选课容量上限（0 表示不限制）
- `time_slot`: 上课时间段（如"周一第1-2节"）

**业务特点**:
- 课程由管理员统一维护
- 支持容量限制，防止超额选课
- 支持时间冲突检测（通过 time_slot 字段）
- 课程可以关联到具体教师，也可以暂时不分配教师（tid 为空）

---

## 5. tb_sct (学生选课表)

**作用**: 记录学生选课关系和成绩信息，是连接学生、课程、教师的核心业务表。

**主要字段**:
- `id`: 主键
- `courseid`: 课程 ID（外键关联 tb_course.id）
- `studentId`: 学生 ID（外键关联 tb_student.id）
- `teaid`: 批改教师 ID（外键关联 tb_teacher.id）
- `score`: 成绩分数
- `graded`: 是否已录入成绩（0=未录入，1=已录入）
- `createtime`: 选课时间

**索引设计**:
- `uk_tb_sct_course_student`: 唯一索引，防止重复选课
- `idx_tb_sct_courseid`: 加速按课程查询
- `idx_tb_sct_studentid`: 加速按学生查询
- `idx_tb_sct_teaid`: 加速按教师查询

**业务特点**:
- 选课时会进行三重校验：
  1. 重复选课检测（通过唯一索引）
  2. 课程容量检测（统计当前选课人数）
  3. 时间冲突检测（检查学生已选课程的时间段）
- 使用悲观锁（SELECT ... FOR UPDATE）防止并发选课导致超额
- 教师只能为自己负责的课程录入成绩
- 成绩录入后会触发通知系统

---

## 6. tb_admin_audit_log (管理员审计日志表)

**作用**: 记录管理员对关键数据的所有操作，用于审计追踪和责任定位。

**主要字段**:
- `id`: 主键
- `admin_id`: 操作管理员 ID
- `admin_username`: 操作管理员账号
- `action`: 操作类型（新增、编辑、删除等）
- `target_type`: 操作对象类型（teacher、student、course、selectionWindow 等）
- `target_id`: 操作对象 ID
- `target_name`: 操作对象名称
- `detail`: 操作详情描述
- `createtime`: 操作时间

**业务特点**:
- 自动记录管理员的所有 CRUD 操作
- 支持按管理员、对象类型、时间范围查询
- 日志只增不改，确保审计完整性
- 用于合规性检查和问题追溯

---

## 7. tb_notification (通知记录表)

**作用**: 记录系统发送给用户的各类通知消息，当前为站内消息系统。

**主要字段**:
- `id`: 主键
- `recipient_id`: 接收人 ID
- `recipient_role`: 接收人角色（admin、teacher、student）
- `recipient_name`: 接收人姓名
- `channel`: 通知渠道（当前统一为 SYSTEM 站内消息）
- `status`: 发送状态（SENT、FAILED 等）
- `title`: 通知标题
- `content`: 通知内容
- `contact`: 联系地址（预留字段，用于未来扩展邮件/短信）
- `result_message`: 结果说明
- `createtime`: 创建时间
- `senttime`: 发送时间

**业务特点**:
- 自动触发通知的场景：
  1. 学生选课成功
  2. 学生退课成功
  3. 教师发布成绩
- 当前仅记录通知，不实际发送邮件或短信
- 预留扩展接口，未来可对接邮件/短信服务

---

## 8. tb_selection_window (选退课时间窗口表)

**作用**: 控制学生选课和退课的时间范围，实现学期选课流程管理。

**主要字段**:
- `id`: 主键
- `action_type`: 操作类型（SELECT=选课，DROP=退课）
- `name`: 窗口名称（如"2026春季学期选课窗口"）
- `start_time`: 开始时间
- `end_time`: 结束时间
- `enabled`: 是否启用（0=停用，1=启用）
- `description`: 窗口说明

**业务特点**:
- 每种操作类型（SELECT/DROP）只能配置一个时间窗口
- 学生只能在窗口开放期间进行选课或退课操作
- 管理员和教师不受时间窗口限制
- 支持启用/停用控制，灵活管理选课流程
- 窗口状态由三个条件共同决定：
  1. `enabled = 1`
  2. 当前时间 >= `start_time`
  3. 当前时间 <= `end_time`

---

## 9. tb_course_prerequisite (课程先修关系表) 🆕

**作用**: 定义课程之间的先修关系，学生必须先修完前置课程才能选修后续课程。

**主要字段**:
- `id`: 主键
- `course_id`: 课程ID
- `prerequisite_course_id`: 先修课程ID
- `min_score`: 先修课程最低分数要求（默认60分）

**业务特点**:
- 一门课程可以有多个先修课程（AND关系，必须全部满足）
- 先修课程必须已修且成绩达到最低要求
- 支持链式先修（A→B→C）
- 典型场景：
  - 数据库原理 需要先修 数据结构（60分以上）
  - 软件架构设计 需要先修 Java Web开发（70分以上）
  - 微服务架构 需要先修 分布式系统（65分以上）

---

## 10. tb_course_mutex (课程互斥关系表) 🆕

**作用**: 定义互斥课程，学生选了A课程就不能再选B课程。

**主要字段**:
- `id`: 主键
- `course_id_a`: 课程A的ID
- `course_id_b`: 课程B的ID
- `reason`: 互斥原因说明

**业务特点**:
- 互斥关系是双向的（A互斥B，则B也互斥A）
- 已选其中一门课程后，不能再选另一门
- 典型场景：
  - Java Web开发 与 Python Web开发 互斥（同类型课程只能选一门）
  - 前端交互设计 与 UI设计方法 互斥（内容重复度高）

---

## 11. tb_course_type_limit (课程类型限选规则表) 🆕

**作用**: 限制学生在某个课程类型下最多可选课程数量。

**主要字段**:
- `id`: 主键
- `course_type`: 课程类型（如"专业必修"、"专业选修"、"通识选修"）
- `max_courses`: 最多可选课程数量
- `description`: 规则说明

**业务特点**:
- 按课程类型限制选课数量
- 防止学生过度选修某一类型课程
- 典型配置：
  - 通识选修：每学期最多3门
  - 专业选修：每学期最多5门

---

## 12. tb_semester_credit_limit (学期学分上限配置表) 🆕

**作用**: 根据学生的学业表现（GPA）动态调整每学期可选学分上限。

**主要字段**:
- `id`: 主键
- `min_gpa`: GPA下限（包含）
- `max_gpa`: GPA上限（不包含）
- `max_credits`: 允许选课的最大学分
- `description`: 规则说明

**业务特点**:
- 根据学生当前GPA确定本学期可选学分上限
- GPA越高，可选学分越多
- 典型配置：
  - GPA < 2.0: 最多15学分（学业预警）
  - 2.0 ≤ GPA < 3.0: 最多20学分（正常）
  - 3.0 ≤ GPA < 3.5: 最多25学分（优秀）
  - GPA ≥ 3.5: 最多30学分（学霸特权）

```

```

## 选课智能冲突检测系统 🆕

### 八重校验逻辑

系统在学生选课时执行以下八重校验（按执行顺序）：

1. ✅ **重复选课检测** - 防止学生重复选同一门课程
2. ✅ **课程容量检测** - 检查课程是否已满员
3. ✅ **时间冲突检测** - 检查上课时间是否与已选课程冲突
4. 🆕 **年级权限校验** - 低年级学生不能选高年级课程
5. 🆕 **先修课程校验** - 必须先修完前置课程且成绩达标
6. 🆕 **互斥课程校验** - 已选其中一门课程后，不能再选另一门
7. 🆕 **课程类型限选校验** - 某类型课程最多只能选N门
8. 🆕 **学分上限校验** - 根据学生GPA动态限制本学期可选学分

### 校验示例

**场景1：先修课程链式校验**
```
学生想选：微服务架构
系统检查：
  ✓ 是否修完 分布式系统？（需要65分以上）
    ✓ 是否修完 数据库原理？（需要60分以上）
      ✓ 是否修完 数据结构？（需要60分以上）
```

**场景2：互斥课程校验**
```
学生已选：Java Web开发
学生想选：Python Web开发
系统拒绝：这两门课程互斥，只能选其一
```

**场景3：学分上限动态调整**
```
学生GPA：3.2
系统允许：最多选25学分
学生已选：18学分
学生想选：8学分的课程
系统拒绝：18 + 8 = 26 > 25，超出学分上限
```

**场景4：年级权限校验**
```
学生年级：大二
学生想选：微服务架构（仅限大四）
系统拒绝：该课程仅限大四及以上年级学生选修
```

## 数据完整性约束

1. **唯一性约束**:
   - 用户名在各自表中唯一（tb_admin、tb_teacher、tb_student）
   - 课程编号唯一（tb_course.numb）
   - 学生不能重复选同一门课（tb_sct 的 courseid + studentId 唯一）
   - 先修关系唯一（tb_course_prerequisite 的 course_id + prerequisite_course_id 唯一）
   - 互斥关系唯一（tb_course_mutex 的 course_id_a + course_id_b 唯一）
   - 课程类型限选规则唯一（tb_course_type_limit 的 course_type 唯一）

2. **业务逻辑约束**:
   - 课程容量限制（max_students）
   - 时间冲突检测（time_slot）
   - 时间窗口控制（tb_selection_window）
   - 🆕 年级权限限制（grade_limit）
   - 🆕 先修课程要求（tb_course_prerequisite）
   - 🆕 互斥课程限制（tb_course_mutex）
   - 🆕 课程类型限选（tb_course_type_limit）
   - 🆕 学分上限控制（tb_semester_credit_limit）

3. **数据一致性**:
   - 选课记录的 teaid 必须与课程的 tid 一致
   - 成绩录入时 graded 字段必须同步更新
   - 🆕 互斥关系是双向的（A互斥B，则B也互斥A）

## 技术亮点 🌟

### 1. 复杂业务逻辑
- **八重嵌套校验**：从简单的重复检测到复杂的链式先修课程校验
- **动态规则引擎**：根据学生GPA动态调整学分上限
- **双向关系处理**：互斥课程的双向查询和校验

### 2. 性能优化
- **悲观锁**：使用 `SELECT ... FOR UPDATE` 防止并发选课导致超额
- **批量查询**：先修课程和互斥课程使用 JOIN 一次性获取
- **索引优化**：为高频查询字段添加索引

### 3. 用户体验
- **友好的错误提示**：每个校验失败都给出明确的原因和当前状态
- **链式校验反馈**：先修课程校验会告知具体缺少哪门课程
- **实时状态展示**：学分上限校验会显示当前已选学分和GPA

## 扩展方向

如需进一步增强系统复杂度，可考虑：

1. **选课优先级算法** - 当课程超员时，按成绩/年级/选课时间自动筛选
2. **课程推荐系统** - 根据已选课程和培养方案推荐最优课表
3. **毕业学分校验** - 自动统计必修/选修/通识学分，判断是否达到毕业标准
4. **智能排课算法** - 自动分析学生空闲时间段，优先安排到空白时间
5. **退课扣分机制** - 超出退课次数后扣除学业分并限制下次选课资格
